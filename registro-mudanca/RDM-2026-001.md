# Registro de Mudança (RDM)

| Campo | Valor |

| **ID da mudança** | RDM-2026-001 |
| **Sistema** | receitas-app |
| **Solicitante** | Cliente (simulado em sala de aula) |

| **Tipo** | Melhoria (alteração de rótulo + nova estrutura de dados) |
| **Prioridade** | Normal |

## Descrição da mudança
1. Alterar o rótulo do título da tela de listagem de receitas, de
   **"Receitas Cadastradas"** para **"Receitos Cadastrados"**.
2. Criar a nova tabela **categoria** (campos `id`, `descricao`) no banco de
   dados, sem afetar os dados já existentes nas tabelas `usuario` e `receita`.

## Justificativa
Demanda de evolução do sistema solicitada pelo cliente, para permitir
futura categorização das receitas cadastradas.

## Impacto
- Código-fonte: `templates/receitas/lista.html`
- Banco de dados: nova migration Flyway `V2__cria_tabela_categoria.sql`
- Novas classes: `Categoria` (model) e `CategoriaRepository`

## Plano de implementação
1. Implementação local (Workspace do desenvolvedor)
2. Versionamento (commit + push para o repositório Git)
3. Integração: testes automatizados (20) + análise de qualidade (SonarQube)
4. Build da aplicação
5. Atualização do ambiente de Homologação
6. Validação em Homologação
7. Atualização do ambiente de Produção
8. Validação em Produção

## Plano de rollback
Reverter o commit no Git e re-executar a pipeline. Como a migration V2 é
aditiva (apenas cria tabela nova), o rollback do código não compromete os
dados já gravados em `categoria`; em caso de rollback completo, a tabela
pode ser removida via migration adicional `V3__remove_categoria.sql`
(não aplicada neste exercício).
